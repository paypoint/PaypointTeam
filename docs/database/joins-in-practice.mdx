# 4. Joins in Practice

Joins are one of the most important and most misunderstood parts of SQL.

In real-world applications, data is rarely stored in a single table.

Instead, data is split into multiple related tables to:

* avoid duplication
* maintain consistency
* improve scalability
* represent real-world relationships

Joins allow us to combine related data from multiple tables and retrieve meaningful information.

The goal is not memorizing syntax — it is learning how relationships work.

---

We will use two tables: `users` and `orders`.

## users

| id | name  | city   |
| -- | ----- | ------ |
| 1  | Rahul | Mumbai |

| 3  | Karan | Mumbai |

| 103 | 2       | 300    |
| 104 | 5       | 1000   |

# 1. Why Joins Exist

Suppose we want:

```text
Customer Name + Order Amount
```

Customer names exist in:

```text
users
```

Order amounts exist in:

```text
orders
```

Since the data is stored in different tables, SQL must combine them.

This is exactly what joins do.

---

# 2. Understanding Relationships

Most joins are based on:

```text
Primary Key
      ↓
Foreign Key
```

Example:

## users

| id |
| -- |
| 1  |
| 2  |

## orders

| user_id |
| ------- |
| 1       |
| 2       |

Relationship:

```text
users.id
      ↓
orders.user_id
```

This relationship tells SQL how records are connected.

---

# 3. INNER JOIN

INNER JOIN returns only matching rows from both tables.

---

## Problem

Get all orders with user names.

```sql
SELECT
    u.name,
    o.amount
FROM users u
INNER JOIN orders o
ON u.id = o.user_id;
```

### Output

| name  | amount |
| ----- | ------ |
| Rahul | 500    |
| Rahul | 800    |
| Anita | 300    |

Notice:

```text
Order 104 is excluded
```

because no matching user exists.

---

# 4. LEFT JOIN

LEFT JOIN returns:

* all rows from left table
* matching rows from right table
* NULL when no match exists

---

## Problem

Get all users and their orders.

```sql
SELECT
    u.name,
    o.amount
FROM users u
LEFT JOIN orders o
ON u.id = o.user_id;
```

### Output

| name  | amount |
| ----- | ------ |
| Rahul | 500    |
| Rahul | 800    |
| Anita | 300    |
| Karan | NULL   |
| Neha  | NULL   |

Users without orders still appear.

---

# 5. RIGHT JOIN

RIGHT JOIN returns:

* all rows from right table
* matching rows from left table
* NULL when no match exists

---

## Problem

Show all orders including orders without valid users.

```sql
SELECT
    u.name,
    o.amount
FROM users u
RIGHT JOIN orders o
ON u.id = o.user_id;
```

### Output

| name  | amount |
| ----- | ------ |
| Rahul | 500    |
| Rahul | 800    |
| Anita | 300    |
| NULL  | 1000   |

Order 104 appears because RIGHT JOIN keeps all rows from orders.

---

# 6. FULL OUTER JOIN

FULL OUTER JOIN returns:

* all rows from left table
* all rows from right table
* matched rows combined
* unmatched rows filled with NULL

---

## Problem

Display all users and all orders.

```sql
SELECT
    u.name,
    o.amount
FROM users u
FULL OUTER JOIN orders o
ON u.id = o.user_id;
```

### Output

| name  | amount |
| ----- | ------ |
| Rahul | 500    |
| Rahul | 800    |
| Anita | 300    |
| Karan | NULL   |
| Neha  | NULL   |
| NULL  | 1000   |

---

# 7. CROSS JOIN

A CROSS JOIN creates every possible combination.

Formula:

```text
Rows in A × Rows in B
```

Example:

```text
4 users × 3 cities
= 12 rows
```

---

## Query

```sql
SELECT
    u.name,
    c.city_name
FROM users u
CROSS JOIN cities c;
```

---

## Common Uses

* Schedule generation
* Matrix reports
* Product combinations
* Testing datasets

---

# 8. SELF JOIN

A table can join itself.

This is called a SELF JOIN.

---

## Example Dataset

employees

| id | name  | manager_id |
| -- | ----- | ---------- |
| 1  | Rahul | NULL       |
| 2  | Anita | 1          |
| 3  | Karan | 1          |

---

## Query

```sql
SELECT
    e.name AS employee,
    m.name AS manager
FROM employees e
LEFT JOIN employees m
ON e.manager_id = m.id;
```

### Output

| employee | manager |
| -------- | ------- |
| Rahul    | NULL    |
| Anita    | Rahul   |
| Karan    | Rahul   |

# 9. Joining Multiple Tables

Joins can connect multiple tables.

Assume:

## payments

| id  | order_id | status  |
| --- | -------- | ------- |
| 201 | 101      | SUCCESS |
| 202 | 102      | FAILED  |
| 203 | 103      | SUCCESS |

---

## Problem

Get user, order amount and payment status.

```sql
SELECT
    u.name,
    o.amount,
    p.status
FROM users u
INNER JOIN orders o
ON u.id = o.user_id
INNER JOIN payments p
ON o.id = p.order_id;
```

### Output

| name  | amount | status  |
| ----- | ------ | ------- |
| Rahul | 500    | SUCCESS |
| Rahul | 800    | FAILED  |
| Anita | 300    | SUCCESS |

---

# 10. JOIN Conditions vs WHERE Conditions

This is one of the most common SQL mistakes.

---

## Correct

Relationship belongs in ON.

Filtering belongs in WHERE.

```sql
SELECT
    u.name,
    o.amount
FROM users u
LEFT JOIN orders o
ON u.id = o.user_id
WHERE u.city = 'Mumbai';
```

### Output

| name  | amount |
| ----- | ------ |
| Rahul | 500    |
| Rahul | 800    |
| Karan | NULL   |

---

## Incorrect

```sql
SELECT
    u.name,
    o.amount
FROM users u
LEFT JOIN orders o
ON u.id = o.user_id
WHERE o.amount > 400;
```

---

### What Goes Wrong?

The WHERE clause removes NULL values.

The LEFT JOIN starts behaving like an INNER JOIN.

Many developers accidentally introduce bugs this way.

# 11. Aggregation with Joins

Joins become much more powerful when combined with aggregation.

---

## Problem

Count total orders per user.

```sql
SELECT
    u.name,
    COUNT(o.id) AS total_orders
FROM users u
LEFT JOIN orders o
ON u.id = o.user_id
GROUP BY u.name;
```

### Output

| name  | total_orders |
| ----- | ------------ |
| Rahul | 2            |
| Anita | 1            |
| Karan | 0            |
| Neha  | 0            |

---

# 12. SUM with Joins

Calculate total spending per user.

```sql
SELECT

FROM users u
LEFT JOIN orders o
ON u.id = o.user_id

```

### Output

| name  | total_spent |
| ----- | ----------- |
| Rahul | 1300        |
| Anita | 300         |
| Karan | NULL        |
| Neha  | NULL        |

# 13. How SQL Executes a Join

When SQL sees:

```sql
SELECT *
FROM users u
INNER JOIN orders o
ON u.id = o.user_id;
```

it roughly performs:

1. Read users table
2. Read orders table
3. Compare join condition
4. Match rows
5. Return combined result

Mental Model:

```text
Find relationship
        ↓
Match rows
        ↓
Return result
```

# 15. Join Performance Basics

Joins become expensive as tables grow.

Performance improves when:

* Primary Keys are indexed
* Foreign Keys are indexed
* Only required columns are selected
* Unnecessary joins are avoided

---

## Bad Practice

```sql
SELECT *
FROM users u
INNER JOIN orders o
ON u.id = o.user_id;
```

---

## Better Practice

```sql
SELECT
    u.name,
    o.amount
FROM users u
INNER JOIN orders o
ON u.id = o.user_id;
```

# 16. Common Join Mistakes

### Missing Join Condition

❌ Wrong

```sql
SELECT *
FROM users, orders;
```

Creates a Cartesian Product.

---

### Joining Wrong Columns

❌ Wrong

```sql
ON users.name = orders.user_id
```

Wrong relationship.

---

### Using INNER JOIN Instead of LEFT JOIN

Can silently hide data.

---

### Filtering in WHERE After LEFT JOIN

May remove NULL rows and change result unexpectedly.

---

# 17. How to Solve Join Problems

Whenever you read a join question:

### Step 1

Identify tables.

### Step 2

Identify relationship.

Example:

```text
users.id = orders.user_id
```

### Step 3

Choose join type.

Need matching rows only?

```text
INNER JOIN
```

Need all users?

```text
LEFT JOIN
```

### Step 4

Apply filters.

### Step 5

Apply aggregation if needed.

---

# 18. Join Summary

| Join Type       | Returns                    |
| --------------- | -------------------------- |
| INNER JOIN      | Matching rows only         |
| LEFT JOIN       | All left rows + matches    |
| RIGHT JOIN      | All right rows + matches   |
| FULL OUTER JOIN | All rows from both tables  |
| CROSS JOIN      | Every possible combination |
| SELF JOIN       | Table joined with itself   |
